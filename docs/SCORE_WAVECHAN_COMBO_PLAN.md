# Score + WaveChan V3 组合方案（完整版）

> 整理日期：2026-03-27
> 状态：**评审完成，待确认方向**
> 参与：Oracle（波浪）、Forge（开发）、Byte（数据）

---

## 一、目标

用 Score 策略选强势股 + WaveChan V3 找回调买点，实现**更稳健的买卖点判断**。

---

## 二、已确认参数

| 项目 | 确认值 |
|------|--------|
| 组合方向 | **先做方向A，方向B暂缓** |
| 调仓频率 | 以信号为准，有信号就操作 |
| WaveChan 信号 | 不限制，W2/W4 都买 |
| 优先买点 | **日线 W2 买点** |
| 数据精度 | 日线（分钟数据暂不可用）|

---

## 三、两种组合方向

### 方向 A：Score 选股 → WaveChan 过滤 ✅ 推荐

```
Score 每日选 Top N 强势股
    ↓
WaveChan 检查是否处于 W2/W4 回调
    ↓
有买点信号 → 买入
```

**优点**：
- 顺势而为逻辑自洽（Score强势股=主升浪）
- 蹭现有Score v4框架，1-2天集成
- 增量成本低，风险小

**风险**：
- W2和W4回调幅度/时间/形态完全不同，等同对待有误判风险
- 日线W2在趋势行情中可能有假信号

### 方向 B：WaveChan 选买点 → Score 过滤 ❌ 暂缓

```
WaveChan 全市场扫描处于 W2/W4 的股票
    ↓
Score 过滤技术面是否强势
    ↓
评分高 → 优先买入
```

**问题**：
- 逻辑顺序反了（回调买点反推强势股）
- 全市场扫描性能瓶颈（30-60分钟，需并行化）
- 如要用，需改为"大盘浪级→WaveChan→Score"三步流程

## 四、团队评审结论

### 各方评审一览

| 维度 | Oracle（波浪） | Forge（开发） | Byte（数据） |
|------|--------------|-------------|-------------|
| 推荐方向 | A | A | A部分支撑，B风险高 |
| 核心理由 | 顺势逻辑自洽 | 1-2天集成，性能可行 | 日线够用，分钟数据缺失 |

### 🔮 Oracle 波浪理论评审

**方向A ✅ 方向B ❌（简单版）**

- Score 强势股 = 主升浪（3浪或5浪），逻辑自洽
- 方向B 用回调买点反推强势股，逻辑顺序反了
- W2 和 W4 回调幅度/时间/形态完全不同，方案等同对待有误判风险
- 方向B 若要用，需改为"大盘浪级→WaveChan→Score"三步流程

**W2/W4 区分问题**：
- 代码中 W2 和 W4 信号未区分
- W2：通常简单（锯齿/平台），回撤可深至78.6%
- W4：通常复杂（三角/联合），回撤通常浅（23.6~38.2%）
- 建议未来补充 Fibonacci 回撤验证

### 🔧 Forge 技术评审

**方向A ✅ 方向B 需改造后上**

- 方向A 蹭 Score v4 框架**完全可行**，1-2天集成
- 主要是串接 `get_snapshot()` 接口
- Score v4 框架本身不需要任何改动
- 方向B 全市场扫描当前需 30-60 分钟
- 用 `SymbolWaveCache` 增量 + `ThreadPoolExecutor(8核)` 可压到 8-15 分钟
- Parquet 仓库完全够用，已有 `warm_cache.py` 渐进预热方案

### 📊 Byte 数据评审

**方向A 部分支撑，方向B 谨慎**

- **日线数据 ✅**：2005-2026，2024年5176只、2025年6148只，覆盖完整
- **分钟数据 ❌**：60分钟线数据严重缺失，WaveChan W2/W4 信号精度大打折扣
- **回测隐患 ⚠️**：
  - 前复权/后复权处理未明确
  - 涨跌停/停牌无过滤（实盘无法执行）
  - look-ahead bias 带来约 +10-12% 虚高
  - Score 22% 年化未经独立审计
- **缠论特殊性**：无预计算的缠论中间结果，每次回测实时计算，效率低+结果不稳定

## 五、综合建议

### 推荐路径

| 优先级 | 行动项 | 负责 | 说明 |
|--------|--------|------|------|
| P0 | 补充 60 分钟数据 | Byte | WaveChan 上线硬约束（暂不可行）|
| P1 | Score v4 → WaveChan 集成 | Forge | 预计 1-2 天，先跑通回测 |
| P2 | 回测 bias 修正 | Byte + Forge | 复权处理 + look-ahead bias + 涨跌停过滤 |
| P2 | 方向B 并行化改造 | Forge | 等方向A 验证后再上 |
| P3 | 三步流程改造方向B | Oracle + Forge | 大盘浪级→WaveChan→Score |

### 下一步建议

1. **先在方向A上跑通回测**，验证逻辑后再投入方向B改造
2. **分钟数据补充是必须项**，否则 WaveChan 信号漂移问题会在实盘中暴露
3. **回测 bias 修正是前提**，否则 Score 22% 年化不可信

## 六、架构设计（方向A）

```
[每日收盘后]
  WaveChan 增量预热 ──> SymbolWaveCache 更新

[每日回测]
  ScoreStrategy.get_signals() ──> 每日 Top N 候选股
                                    ↓
                        WaveEngine.get_snapshot() ──> W2/W4 状态检查
                                    ↓
                        买点信号过滤 ──> 最终买入信号
```
