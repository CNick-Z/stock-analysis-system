## 1. RSI14阈值 → 修改为：分层预警（推荐）

### 问题1：SPEC 用 RSI<40，但原始研究显示 RSI<50 更可靠

**事实核查：**

| 来源 | 结论 |
|------|------|
| `V8_IC_IR_TUNING.md` | RSI 超卖区间（<25~35）是**陷阱**，越超卖越差 |
| `V8_ICIR_OPTIMIZATION_20260330.md` | RSI ∈ [50,60) 区间胜率 43-44%，最优 |
| V8 策略参数 | 入场过滤：RSI 50-65（18年验证） |
| SPEC 声称 | RSI<40 = 熊市信号（仓上限 30%） |

**分析：**
- RSI<40 是严重超卖区域，此时市场往往已经大幅下跌
- 纯从预警角度，RSI<50 确实能更早发出信号（领先 15-28 天，如任务描述）
- 但纯 RSI<40 会减少误判——只在真正弱势时触发

**结论：分层预警更合理**
```
RSI < 40 → BEAR（确认熊市），仓位上限 30%
RSI < 50 → NEUTRAL（预警熊市），仓位上限 60%
```

### 问题2：RSI 数据不足 30 天时标记 BULL —— 合理

**评审意见：✅ 同意保守处理**

理由：
- 30 天窗口足够覆盖 2×RSI 周期，数据不足时默认 BULL 是合理的安全边际
- 标注为"保守处理"且对外透明，符合 SOUL.md 的"不隐瞒风险"原则

## 2. 仓位参数 → 修改为：分层降仓（RSI<50→60%，RSI<40→30%）

### SPEC 现有逻辑的问题

| 阈值 | SPEC | 建议 |
|------|------|------|
| RSI<40 | 30% | 维持 |
| MA5<MA10<MA20 + MACD<0 | 70% | 这两个条件**不对齐**——建议改为 RSI<50 |

**核心逻辑缺陷：**
SPEC 中 Trader 补充的 NEUTRAL 条件与主条件（MA20 + RSI<40）逻辑不兼容：
- 如果 RSI<40，position_limit 已直接设为 30%，NEUTRAL 分支根本不会触发
- NEUTRAL 分支实际上是一个**独立判断**，只在 `not (close<MA20 AND RSI<40)` 时生效
- 这意味着 NEUTRAL 是在"非熊市确认"的情况下，用另一套条件重新判断为偏熊

**建议修正逻辑：**
```
IF close < MA20 AND RSI < 40:
    regime = BEAR; position_limit = 0.30; signal = "ma20_bear_confirmed"
ELIF RSI < 50:  # 提前预警，不等跌破 MA20
    regime = NEUTRAL; position_limit = 0.60; signal = "rsi_warning"
ELIF MA5 < MA10 < MA20 AND MACD_hist < 0:  # 仅作为二次确认
    regime = NEUTRAL; position_limit = 0.70; signal = "trend_bear_confirmed"
ELSE:
    regime = BULL; position_limit = 1.00; signal = "bull"
```

## 3. 其他修改建议

### 🔴 严重问题：回测数据无法核实

**SPEC 引用数据：**
> "年化 +1.6% → +25.5%，夏普 0.07 → 1.46，最大回撤 -47% → -22%"

**实际 V8 18年回测数据（`RESULT_V8_18YEAR_BACKTEST.md`）：**
- V8 基准：跨17年平均年化 **+5.28%**，夏普 **0.45**
- SPEC 的 +25.5% 年化在 18 年数据中**从未出现**
- SPEC 的基准 +1.6% 也不是整体平均（整体为 +5.28%）
- 最大回撤 -47% 可能来自某个特定年份（如 2011 的 -35.4%，或更早期）

**问题：**
1. SPEC 声称的 +25.5% 年化没有对应回测报告支撑
2. "年化 +1.6%" 的基准时间段不明确
3. Fairy 在编写 SPEC 时引用了不存在或未经 Oracle 确认的数据

**建议：** 在回测验证完成前，将 SPEC 的回测数据标注为**"待核实"**，以实际回测结果为准。**数据来源必须可追溯。**

### 🟡 建议1：增加 NEUTRAL 状态的实际触发条件

SPEC 中 NEUTRAL 的 second confirmation 条件（MA5<MA10<MA20 + MACD<0）与主 BEAR 判断独立，这会导致：
- 主条件触发 BEAR（RSI<40）时，NEUTRAL 永远不触发
- 实际上 NEUTRAL 变成了一个"非 BEAR 情况下的偏熊判断"

建议明确定义 NEUTRAL 的独立触发条件（建议用 RSI<50 作为主要条件）。

### 🟡 建议2：三窗口回测时间段需要扩展验证

| 窗口 | SPEC 定义 | 问题 |
|------|----------|------|
| 窗口1 | 2024（牛市） | 仅1年，样本不足 |
| 窗口2 | 2022-2023（熊市） | ✅ 合理 |
| 窗口3 | 2020-2021（震荡） | ✅ 合理 |

建议增加：
- 2010-2012（A股历史大熊市，三年连跌）
- 2015-2016（股灾回测）
- 总计覆盖至少 5 个完整牛熊周期

### 🟡 建议3：仓位计算逻辑需要明确

SPEC 中 `effective_position_size = min(原始position_size, 大盘仓位上限)` 存在逻辑问题：
- 如果 position_size = 15%（单笔仓位），limit = 30%，则实际 = min(15%, 30%) = 15%，**没有降仓效果**
- 正确的降仓应该是：`可用资金 = 现金 × limit`，然后分配给各持仓

建议明确定义 position_limit 的语义：
```
方案A（推荐）：position_limit = 30% 表示总仓位上限为账户的 30%
方案B：单笔仓位 = 原始单笔 × position_limit（即 min 的用法）
```

### 🟢 建议4：信号标签不一致

SPEC 定义了：
- `signal = "ma20_bear"` → 主 BEAR 信号
- `signal = "ma10_bear_confirmed"` → NEUTRAL 信号（标注为"Trader补充"）

建议统一标签格式：`{trigger}_{regime}` 格式，如 `rsi_bear`、`ma20_bear`、`trend_bear_confirmed`。

## 4. 总体评价 → 🔴 需修改后通过

### 核心问题（必须解决）

| 优先级 | 问题 | 状态 |
|--------|------|------|
| 🔴 P0 | 回测数据无法核实（+25.5%年化无来源） | 必须在回测验证前移除或标注"待核实" |
| 🔴 P0 | 仓位降仓逻辑不明确（min(position_size, limit) 无实际降仓效果） | 必须明确语义 |
| 🟡 P1 | RSI<40 阈值过严，建议分层预警（RSI<50 预警，RSI<40 确认） | 强烈建议修改 |

### 通过条件

1. ✅ 回测数据必须来源于实际回测报告，标注时间段和回测引擎
2. ✅ 仓位计算逻辑必须明确并通过单元测试验证
3. ✅ RSI 分层预警逻辑落地后，用三窗口完整回测验证
4. ✅ NEUTRAL 触发条件必须独立且可触发（建议 RSI<50 作为独立触发条件）

### 结论

SPEC 的**设计方向正确**（大盘牛熊过滤确实是 V8 熊市回撤的核心解决方案），但**参数选择和数据来源存在严重问题**。Oracle 无法在数据未核实的情况下背书 +25.5% 年化这一关键数据。

建议 Fairy 协调重新执行回测，Oracle 提供 RSI 分层阈值方案，Trader 确认仓位参数后，再交由 Forge 实现。
