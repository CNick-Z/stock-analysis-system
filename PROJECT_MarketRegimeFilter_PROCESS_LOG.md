### 阶段六：position_limit 逻辑修复（11:53 - 11:56）

| 时间 | 事项 | 负责人 |
|------|------|--------|
| 11:53 | Oracle 发现根本问题：position_limit 与 position_size 用 min() 混用 | Oracle |
| 11:54 | Forge 修复：移除错误的 min() | Forge |
| 11:56 | 验证通过，SPEC 更新至 v2.3 | Forge |

**修复内容：**
```python
# 修复前（错误）：
effective_position_size = min(self.position_size, position_limit)
avail_cash = self.cash * effective_position_size

# 修复后（正确）：
avail_cash = self.cash * position_limit  # 受 regime 控制
per_stock_cash = min(avail_cash / fill_count, self.cash * self.position_size)  # 均分后再受单股上限
```

| 参数 | 原值 | 新值 |
|------|------|------|
| confirm_days | 2 | **1** |
| NEUTRAL position_limit | 1.00 | **0.70** |

**三方关键洞察：**
- **Trader**：confirm_days=2 信号延迟在实盘无法弥补；最大回撤发生在"买在高点之后"，position_limit 管不了已持仓亏损
- **Oracle**：NEUTRAL=1.0 是5/6窗口失效的核心原因；confirm_days=1 假信号增量有限（6-12天/年）
- **Forge**：加状态锁防抖动；保留模块但默认不启用，作为未来大熊市快速启用选项

## 🔑 重点发现

### 1. 代码评审发现了什么
- **Trader 发现 P0**：Fallback 默认满仓（100%），实盘极度危险——三方评审中唯一一个真正从实盘视角出发的问题
- **教训**：三方评审都在检查"模块对不对"，没有一个人真正 trace 资金流

### 2. Oracle 回测发现了什么
- **position_limit 与 position_size 用 min() 混用**——导致过滤器永远不生效
- 这是开发时的语义错误，不是参数问题

### 3. 第二次回测发现了什么
- **参数设计问题**：NEUTRAL=100% 导致过滤器在多数时间等于废的
- 过滤器只在 Window F（26.5% BEAR天数，极端长熊市）才有效

### 4. 三方讨论的共识
- 放弃过滤器不放弃思路：这套框架在研究层面有价值
- V8 本身的风控（入场前算好单票最大亏损≤2% + 持仓后跟踪止损）可能比大盘过滤器更有效

## 📊 v2.5 回测结果（2026-04-02 13:35）

### 最终结果

| 窗口 | 无过滤 MaxDD | v2.5有过滤 MaxDD | 改善 | 无过滤年化 | v2.5有过滤年化 | 年化变化 |
|------|-------------|-----------------|------|----------|--------------|---------|
| **F** (2010-12 三年熊) | -45.02% | **-36.86%** | **+8.16pp** ✅ | -12.81% | **-9.75%** | **+3.06pp** ✅ |
| **B** (2022-23 熊市) | -10.42% | **-8.30%** | **+2.12pp** ✅ | +9.64% | **+7.47%** | -2.17pp ⚠️ |
| **E** (2018 熊市) | -23.28% | **-18.91%** | **+4.37pp** ✅ | -5.76% | **-5.58%** | **+0.18pp** ✅ |

### 结论

- **3个窗口全部有效**，MaxDD 全面改善
- Window F 效果最显著（BEAR天数占比32.6%）
- Window B 有代价：年化-2.17pp，但换来 MaxDD -2.12pp（NEUTRAL降仓影响了正收益）
- **position_limit 叠加公式修复是真正的根因解决**

### 教训总结（完整版）

1. **第一次 bug**：`min(position_size, position_limit)` 让过滤器永远不生效
2. **第二次 bug**（发现于 v2.3）：移除 min 后，`per_stock` 仍然被 `position_size` 的 min() 抵消
3. **根因**：三次 bug 都源于对 `position_limit` 语义的理解不准确——它是 `position_size` 的**叠加限制器**，不是独立的资金上限
4. **资金流必须 trace**：三次 debug 都是通过逐行 trace 资金流发现的
