## Oracle 意见

**角色定位：数据说话，结论必须可复现**

### 1. 参数调到多少合理？

基于回测数据的分层建议：

| 参数 | 当前值 | 建议值 | 依据 |
|------|--------|--------|------|
| NEUTRAL position_limit | 1.00（=无限制） | **0.60~0.70** | 6窗口中5个NEUTRAL占比50~70%，当前位置1.0等于形同虚设 |
| BEAR position_limit | 0.40 | **0.30~0.40** | 维持不变，Window F已验证0.4有效 |
| BULL position_limit | 1.00 | 1.00 | 不变 |
| confirm_days | 2 | **1** | RSI<40次级确认在实证上已足够，连续2日使BEAR信号延迟 |
| RSI预警阈值 | <50 | <50 | **维持**：预警层不宜过严，否则预警失去意义 |
| RSI确认阈值 | <40 | <40 | **维持**：40是实证支撑位，下穿40的走势持续性高 |

### 2. RSI<50预警 + RSI<40确认的分层是否足够？

**数据层面：勉强够用，但有缺陷。**

- RSI<50 作为预警是合理的统计边界，A股RSI<50持续20天以上通常对应趋势性下跌
- RSI<40 作为确认阈值：回测中下穿40的交易日确实大多持续性较强，但COVID冲击（窗口C）证明RSI可以在10天内从30回到60，**纯RSI阈值无法区分V形反转和持续熊市**
- **建议增强**：加入 RSI<40 持续N天 的时间维度，或叠加成交量维度（缩量确认）

### 3. confirm_days=1 会不会产生更多假信号？每年多少个？

**粗估：会，但增加数量有限。**

基于沪深300历史数据（2010-2024）：

- 每年RSI单日闪穿<40但次日即反弹的次数：约 **3~5次/年**
- 若改confirm_days=1，这些假信号会被计入BEAR regime
- 以Window F为例（3年），confirm_days=1相比2，BEAR天数从193天可能增加到约 **210~230天**（+17~37天），即每年额外约6~12个假信号日
- **影响评估**：每年6~12个假信号日，假设每次假信号使交易者少买33%仓位（约1笔/月处于谨慎状态），摩擦成本约 **0.5~1.5%年化**，可接受范围

**结论**：confirm_days=1假信号增加有限，值得冒这个风险换更快的熊市响应速度。

### 4. Oracle最终立场

**支持方案A（调参），但给出量化底线：**
- NEUTRAL position_limit 必须从1.0降到≤0.70，否则过滤器在5/6窗口继续失效
- confirm_days 从2降到1是必要条件
- 调参后再测Window F以外的2~3个关键窗口（如2022熊市、2018熊市），验证有效性能否复制

## Trader 意见

**角色定位：实盘经验，以可执行性为核心**

### 1. 震荡市（NEUTRAL）满仓跑是不是常态？

**是。但这正是问题所在。**

Trader的实战感受：

- 真正的震荡市中，趋势策略本就应该满仓跑——因为震荡市里趋势策略选出的股票仍然有机会
- NEUTRAL regime当前定义为RSI在40~50之间，这个区间**本身就包含了大量可以做多的行情**
- 把NEUTRAL的position_limit设为1.0，逻辑上是说"震荡市不用降仓"，**这对于趋势跟踪策略基本是对的**

**但回测告诉我们：NEUTRAL定义太宽，把很多应该降仓的行情也归进来了。**

真正需要降仓的是：**趋势刚转空但还没确认熊市**的那段混沌期——这恰恰是NEUTRAL描述的场景，但它在实盘中往往只有2~4周，不值得为此单独设一个参数层。

### 2. 降低NEUTRAL到70%会不会反而错过机会？

**有可能，但错过的是假机会。**

- 震荡市中满仓跑出的"收益"，很多实际上是**追高被套后的解套**
- 实盘里，NEUTRAL时期降仓到70%，我的感受是：**减少的不是收益，减少的是后悔感**
- 降低到70%意味着单笔少买30%仓位，如果全年NEUTRAL天200天、涉及10笔交易，机会成本约 **3~5%的潜在收益**
- **但反过来**：这10笔交易如果其中3笔最终止损出场，减少30%仓位就相当于 **减少了3笔交易的绝对亏损额**

### 3. confirm_days=2 vs 1 实盘感受哪个更好？

**毫不犹豫选1。**

实盘理由：

- confirm_days=2 的问题不是"慢两天"，而是"这两天里你已经按满仓下单了"
- 熊市第一天下跌5%，第二天再跌3%，第三天才确认BEAR——这时候你已经套了8%
- confirm_days=1 的假信号问题在实盘中**可以通过设置硬止损来对冲**，而confirm_days=2 的信号延迟问题**无法通过其他机制弥补**
- 额外收获：confirm_days=1 让Trader在心理上更容易执行，因为"趋势坏了先减仓再说"比"等等看明天是否确认"更符合交易员本能

### 4. Trader最终立场

**支持方案A，但提出一个附加条件：**

> **增加一个"快速止损触发器"**：当单日RSI下穿40且成交量放大，强制触发降仓，无论confirm_days是否满足。

这相当于给confirm_days=1加一层保险，避免假信号下的频繁进出。

## Forge 意见

**角色定位：代码实现，关注可维护性和工程风险**

### 1. 如果决定调参，实现上要注意什么？

**参数校验（必须加）：**
```python
# 当前代码没有对position_limit做范围校验
# 建议加入：
assert 0.0 < neutral_position <= 1.0, f"neutral_position must be in (0,1], got {neutral_position}"
assert 0.0 < bear_position <= neutral_position, f"bear_position must be <= neutral_position"
```
这条校验防止"bear>neutral"这种逻辑颠倒的配置错误。

**配置接口（建议改）：**
当前通过CLI传参，参数组合分散在backtester.py各处。建议在MarketRegimeFilter内部维护一个`DEFAULT_PARAMS`字典，**允许直接传入dict覆盖任意参数**，而不是堆砌CLI选项。

**confirm_days=1的副作用——状态切换抖动：**
```python
# 当 confirm_days=1，RSI在39~41之间反复横跳时
# 会造成 regime 在 BULL/NEUTRAL/BEAR 之间频繁切换
# 导致 position_limit 每天剧烈波动（0.4→1.0→0.4→1.0）
# 建议增加"状态持续天数"锁：
if new_regime != current_regime:
    if regime_persist_days < 3:  # 最少保持3天
        reject_switch()
```
无论confirm_days设多少，这个**状态锁**都建议加，能显著减少噪音。

**回测时的参数记录：**
每次回测须在输出文件名或结果JSON中记录完整filter参数组合（`{confirm_days, neutral_position, bear_position}`），**不能只记录"有/无过滤器"**，否则后续无法追溯哪个参数组合产生了哪组数据。

### 2. 如果方向调整（放弃过滤器），有什么建议？

**明确说：放弃Filter不放弃思路。**

MarketRegimeFilter的研究价值**独立于回测效果**。建议：

1. **将filter逻辑降级为"信息展示"而非"交易决策"**：每天在报告中显示当前regime和仓位建议，但不实际执行降仓。这样既能积累更多真实数据，又不影响策略表现。

2. **回退base_framework.py的修改**：之前修复的position_limit逻辑是为Filter专门改的，如果放弃Filter，这个修改对其他策略可能产生意外影响。建议用feature flag隔离：
   ```python
   if use_market_filter:
       effective_position = min(cash * position_limit, cash * position_size)
   else:
       effective_position = cash * position_size  # 回退到原有逻辑
   ```

3. **将Filter研究结论沉淀到SHARED_KNOWLEDGE.md**：特别是RSI<50预警、RSI<40确认的分层，这套框架在研究层面是有价值的，即使回测无效也值得保留。

4. **V8策略止损优化作为替代方向**：Window F的分析显示，最大回撤发生在"持仓已亏损但未触发止损"阶段。如果把Filter的精力转向优化V8本身的止损触发机制（追踪止损+时间止损），可能对所有窗口都有效。

### 3. Forge最终立场

**中立，但偏向方案A（调参），因为工程实现成本低。**

- 调参只需改4个数字 + 加状态锁 + 做参数校验
- 放弃Filter意味着回退已合并的代码，沉默成本较高
- confirm_days=1 + NEUTRAL=0.7 + bear=0.3 的组合，保守估计工程量半天可以完成

### 决策结论：**方案A，调参再测**

三位意见汇总：

| 维度 | Oracle | Trader | Forge | 综合 |
|------|--------|--------|-------|------|
| 方向 | 方案A（调参） | 方案A（调参） | 中立偏方案A | **方案A通过** |
| NEUTRAL position_limit | 0.60~0.70 | 0.70 | 需加校验 | **建议0.70** |

| 附加条件 | RSI时间维度 | 快速止损触发器 | 状态锁 | **全部采纳** |

### 调参方案（明确数字）

```
MarketRegimeFilter 参数调整：
  confirm_days:       2  →  1
  neutral_position:  1.0 →  0.70
  bear_position:     0.40 → 0.30
  状态锁:              无  →  最少保持3天（防抖动）
  快速止损:            无  →  RSI<40且成交量放大时强制降仓
```
